# Анализ и оптимизация (v1.1.0)

Дата: 2026-07-11

## Что было найдено

### Критично (исправлено)

| Проблема | Влияние | Решение |
|----------|---------|---------|
| Поиск вызывал `Get-NetFirewallRule` на каждую букву | Лаги UI, нагрузка на систему | Разделение `Reload` / `Apply-RuleFilters` + debounce 300ms |
| CLI дублировал логику Core (~200 строк) | Расхождение GUI/CLI при изменениях | CLI переведён на `FirewallManager.Core.ps1` |
| Двойной refresh при клике на карточки | Лишние перерисовки | Флаг `FilterSync` |

### Средний приоритет (исправлено)

| Проблема | Решение |
|----------|---------|
| 4× `Where-Object` в статистике | `Get-FirewallRuleStats` — один проход |
| Создание Brush при каждом выборе | Кэш `BrushSelected` / `BrushMuted` |
| Дублирование логики фильтрации | `Get-FirewallRulesFiltered` в Core |
| Мёртвая переменная `hasPortProto` | Удалена |

### Низкий приоритет (безопасно отложено)

| Идея | Почему не сейчас |
|------|------------------|
| Тёмная тема | Новые стили, риск регрессии |
| Toast вместо MessageBox | Новый UI-компонент |
| Редактирование правил | Новая функция, не оптимизация |
| Bulk CIM-запрос фильтров | Сложность, выигрыш только при 500+ правилах |

## Архитектура после оптимизации

```
FirewallManager.Core.ps1   ← единый API
        ↑           ↑
   GUI.ps1      CLI.ps1
```

### GUI: поток данных

1. `Reload-RulesFromSystem` — читает брандмауэр (F5, CRUD, старт)
2. `Apply-RuleFilters` — фильтрует в памяти (поиск, вкладки)
3. `SearchDebounce` — 300ms задержка при вводе
